Postgres equality is case sensitive by default

So if you make this query, SELECT * FROM a_table WHERE a_column LIKE '%some text%';, you get 1 result back.

However, if you change your pattern into upper case, SELECT * FROM a_table WHERE a_column LIKE '%SOME TEXT%';, you get 0 result back.

Solutions...

  1. Use CITEXT datatype.

    Example: In a migration script, do change_column :a_table, :a_column, :citext

  2. If a migration is not desirable, use the ILIKE keyword instead.

    Back to the a_table example, if you do the following query, you get 1 result back.

    SELECT * FROM a_table WHERE a_column ILIKE '%SOME TEXT%';

But...

Neither solutions work well with B-tree indexes. If you need to do a case insensitive query against a large table, you are screwed.

  1. When you are using a CITEXT datatype, value comparison does not utilize index because it is doing the following internally:

    SELECT * FROM tab WHERE lower(col) = LOWER(?);

    (reference from http://www.postgresql.org/docs/9.3/static/citext.html)

  2. When you are using the ILIKE keyword, postgres utilizes the index only if the pattern starts with non-alphabetic characters. So it is pretty useless.

    (reference from http://www.postgresql.org/docs/9.2/static/indexes-types.html)

So to unscrew it...

You will need to use a functional index. Below is the example of utilizing a lower case functional index against our search_events partition tables.

  1. Create a lower case functional index.

    CREATE INDEX a_table_lower_a_column_idx ON a_table (lower(query) varchar_pattern_ops);

  2. Remember to lower everything in your scope.

    where('lower(a_column) = ?', 'SOME TEXT'.downcase)

  3. Hope that you remember to down case everything 6 months later.
